# 고밀도 관리자 웹 UI 제작 지시서

이 문서는 다른 AI 또는 개발자에게 웹 관리자 페이지, 회원가입, 신청서, 상담 신청서 화면을 만들라고 지시할 때 사용하는 기준 문서다.

---

## 0. 최종 목표

웹 화면은 넓고 여백 많은 디자인이 아니라 **업무용 고밀도 관리자 UI**로 만든다.  
한 화면에서 많은 정보를 빠르게 보고, 클릭 즉시 상세를 확인하고, 필요한 관리 버튼을 바로 사용할 수 있어야 한다.

핵심 문장:

> DB 테이블의 한 칸은 가로로 여러 정보를 나열하는 것이 아니라, 하나의 셀 안에 관련 정보 1~3개를 세로로 쌓아 보여준다. 반복되는 단어는 이모지, 아이콘, 배지로 줄인다.

---

## 1. 전체 레이아웃 원칙

### 1-1. 고밀도 UI

- 화면은 촘촘하게 구성한다.
- 불필요한 큰 여백, 큰 카드, 과한 줄 간격을 피한다.
- 관리자 화면은 한눈에 많은 정보를 봐야 하므로 테이블, 버튼, 상태값, 상세 패널을 압축적으로 배치한다.

권장 글자 크기:

| 용도 | 크기 |
|---|---:|
| 큰 제목 | 13~18px |
| 일반 본문/테이블 | 10~11px |
| 보조 정보 | 9~10px |
| 아주 작은 상태/라벨 | 8~9px |

---

## 2. 상단 탭 규칙

DB 관리자 화면 상단에는 DB 관점별 탭을 둔다.

### 탭 이름 규칙

`보기`라는 단어를 붙이지 않는다.

잘못된 예:

- 전체보기
- 회원보기
- 결제현황보기
- 상담현황보기

올바른 예:

- 전체
- 회원
- 결제현황
- 상담현황
- 응시현황
- 수수료현황
- 신청서
- 관리자메모

---

## 3. DB 테이블 셀 구성 규칙

### 3-1. 가장 중요한 규칙: 한 칸에 세로로 1~3개 정보

테이블 한 칸에는 관련 정보를 **가로가 아니라 세로로** 넣는다.

잘못된 예:

```text
입준혁 | 39 | 남 | 전문가 | 프리미엄
```

올바른 예:

```text
입준혁
39 ♂
```

```text
junhyuk.lim@email.com
010-9901-1287
```

```text
서울 관악구
🇰🇷 대한민국
```

```text
🧑‍🏫 전문가
💎 프리미엄
```

### 3-2. 셀별 추천 묶음

| 컬럼 | 셀 안에 세로로 넣을 정보 |
|---|---|
| 이름 / 나이 | 이름, 나이+성별 아이콘 |
| 연락처 | 이메일, 휴대폰 |
| 주소 | 시/구, 국가 아이콘 |
| 유형 / 구독 | 회원유형 이모지, 구독등급 이모지 |
| 상태 / 등급 | 활성/대기/비활성 배지, A/S/B 등급 |
| 가입 / 로그인 | 가입일, 최근 로그인 |
| 매칭 / 상담 | 매칭 수, 상담 수 |
| 결제 | 금액, 완료/미납 상태 |
| 메모 | 핵심 메모, 보조 메모 |

---

## 4. 반복 단어 이모지/배지 변환 규칙

반복되는 단어는 계속 텍스트로 쓰지 말고 이모지나 짧은 배지로 바꾼다.

| 반복 단어 | 표시 방식 |
|---|---|
| 남자 | ♂ |
| 여자 | ♀ |
| 기타/미선택 | ⚧ |
| 일반회원 | 👤 일반 |
| 전문가회원 | 🧑‍🏫 전문가 |
| 기업/단체 | 🏢 기업 |
| 프리미엄 | 💎 프리미엄 |
| 스탠다드 | 🔷 스탠다드 |
| 무료 | 🆓 무료 |
| 활성/온라인 | 🟢 활성 |
| 대기 | 🟡 대기 |
| 비활성/오프라인 | 🔴 비활성 |
| 결제완료 | ✅ 완료 또는 💳 완료 |
| 미납 | ⚠️ 미납 |
| 상담 | 💬 |
| 전화 | 📞 |
| 이메일 | ✉ |
| 문자/SMS | 📨 |
| 국가 대한민국 | 🇰🇷 대한민국 |
| 전문가 레벨 | Lv.8 같은 짧은 배지 |

주의:

- 이모지는 정보를 줄이기 위한 보조 수단이다.
- 너무 많은 이모지를 한 셀에 넣어 장식처럼 보이게 하지 않는다.
- 반복되는 정보에만 우선 적용한다.

## 5. DB 테이블 + 상세 패널 동작 규칙

### 5-1. 선택 전

- 테이블이 화면 전체 너비를 꽉 채운다.
- 오른쪽에 빈 상세 공간을 미리 남겨두지 않는다.

### 5-2. 선택 후

- 사용자가 테이블의 어느 셀을 클릭해도 해당 행이 선택된다.
- 선택되면 테이블이 왼쪽으로 줄어든다.
- 오른쪽에 상세 패널이 열린다.
- 상세 패널 위치는 오른쪽 고정이다.

구조:

```text
선택 전:
[ DB 테이블 100% ]

선택 후:
[ DB 테이블 축소 ] [ 오른쪽 상세 패널 ]
```

### 5-3. 체크박스와 클릭 구분

- 체크박스: 여러 명 선택 후 일괄 처리용
- 행/셀 클릭: 상세 패널 열기용

---

## 6. 오른쪽 상세 패널 구성

상세 패널은 오른쪽에 열린다. 내부는 2단 또는 4단 그리드로 촘촘하게 구성한다.

필수 영역:

1. 상단 프로필
   - 이름
   - 나이/성별
   - 회원유형/구독상태
   - 상태/등급
2. 상세 탭
   - 기본정보
   - 상담/매칭
   - 결제/수수료
   - 활동/이력
   - 메모
3. 4단 정보 그리드
   - 이메일
   - 휴대폰
   - 가입일
   - 최근로그인
   - 주소
   - 생년월일
   - 전문분야
   - 회원상태
4. 요약 카드
   - 상담 요약
   - 최근 상담 로그
   - 결제 요약
   - 관리자 메모
5. 하단 버튼
   - SMS 전송
   - 이메일 발송
   - 상담 변경
   - 메모 작성
   - 저장
   - 닫기

---

## 7. 버튼 배치 원칙

버튼은 최소화하지 않는다.  
**필요 없는 버튼만 제외하고, 관리에 필요한 버튼은 기본적으로 노출한다.**

관리자 화면 기본 버튼 예시:

- 추가
- 수정
- 삭제
- 저장
- 닫기
- 더보기
- 상태변경
- 등급변경
- 상담변경
- 결제확인
- SMS 전송
- 이메일 발송
- 내보내기
- 메모 작성
- 선택 삭제
- 선택 내보내기

단, 실제로 해당 화면에서 의미 없는 버튼은 제외한다.

## 8. 회원가입 화면 규칙

회원가입 화면도 한 줄에 무리하게 펼치지 않는다.  
관련 있는 항목끼리 묶고, 한 묶음 안에서 1~3개 정보를 세로/그리드로 정리한다.

### 기본 구조

1. 기본정보
   - 이름
   - 성별/나이
   - 생년월일
2. 연락정보
   - 이메일
   - 휴대폰
   - 비밀번호
3. 회원구분
   - 👤 일반
   - 🧑‍🏫 전문가
   - 🏢 기업
   - 💎 프리미엄
4. 약관동의
   - 전체 동의
   - 세부 약관 제목
   - 펼치기 버튼
5. 가입/취소 버튼

---

## 9. 신청서 화면 규칙

신청서는 세로로 길게 나열하지 않는다.  
같은 성격의 항목끼리 묶어서 사용자가 보기 편하게 만든다.

### 신청서 추천 구조

1. 신청 분류
   - 지원구분
   - 과정선택
   - 세부과정
   - 신청유형
2. 신청자 정보
   - 이름
   - 연락처
   - 이메일
3. 과정 요약
   - 기간
   - 시간
   - 수강료
   - 잔여석
4. 학습 목표 / 참여 동기
5. 전체 동의
6. 이전/신청하기 버튼

---

## 10. 문의/상담 신청서 규칙

문의/상담 신청서는 가독성을 우선한다.  
과한 주석, 빨간 표시, 장식은 줄인다.

추천 구조:

1. 문의 분류
   - 문의구분
   - 과정선택
   - 우선순위
2. 연락 정보
   - 이름
   - 연락처
   - 이메일
3. 문의 내용
4. 전체 동의
5. 취소/임시저장/문의하기 버튼

---

## 11. 전체동의 영역 규칙

### 11-1. 기본 상태

- 전체 동의는 기본 체크되어 있어야 한다.
- 세부 약관도 기본 체크되어 있어야 한다.
- 접힌 상태에서는 약관 본문 내용을 보여주지 않는다.
- 접힌 상태에서는 제목과 체크박스만 보인다.

접힌 상태 예시:

```text
☑ 전체 동의합니다                         [펼치기]
☑ 서비스 이용약관
☑ 개인정보 수집 및 이용
☑ 마케팅 정보 수신 동의
```

### 11-2. 펼친 상태

- 펼치기를 눌렀을 때만 약관 본문이 보인다.
- 본문은 별도 박스 안에 표시한다.
- 본문이 길면 내부 스크롤을 쓴다.

---

## 12. 주석/설명 표시 규칙

목업에는 핵심 원칙 설명이 필요하지만 너무 어수선하면 안 된다.

- 빨간 점, 점선 박스, 화살표는 핵심 3~5개만 사용한다.
- 실제 사용 화면에는 주석을 넣지 않는다.
- 설명용 목업에만 주석을 넣는다.

강조해야 할 핵심 포인트:

1. 한 칸에 세로로 1~3개 정보
2. 반복 단어는 이모지/배지로 축약
3. 선택 전 테이블 100%, 선택 후 테이블 축소 + 오른쪽 상세
4. 신청서는 같은 항목끼리 묶기
5. 전체동의는 접힌 상태에서 제목만 표시

## 13. 개발용 HTML/CSS 핵심 클래스 예시

### 테이블 셀 세로 묶음

```html

    <span class="top">입준혁</span>
    <span class="sub">39 ♂</span>
  </div>

```

.cellstack {
  display: flex;
  flex-direction: column;
  gap: 2px;
  justify-content: center;
}
.cellstack .top {
  font-weight: 800;
}

  font-size: 10px;
  color: #374151;

```

.db-wrap {

  grid-template-columns: minmax(680px, 1.08fr) minmax(430px, .92fr);
  gap: 8px;
}
```

## 14. 다른 AI에게 줄 최종 지시문

아래 문장을 그대로 다른 AI에게 전달하면 된다.

```text
고밀도 관리자 웹 UI를 만들어줘. 전체 화면은 촘촘하게 구성하고, DB 화면 상단 탭은 전체/회원/결제현황/상담현황/응시현황/수수료현황/신청서/관리자메모처럼 짧게 쓰고 ‘보기’라는 단어는 빼줘.

DB 테이블에서 가장 중요한 점은 한 칸에 관련 정보 1~3개를 가로가 아니라 세로로 쌓아 보여주는 거야. 예를 들어 이름 칸은 ‘입준혁 / 39 ♂’, 연락처 칸은 ‘이메일 / 휴대폰’, 주소 칸은 ‘서울 관악구 / 🇰🇷 대한민국’, 회원유형 칸은 ‘🧑‍🏫 전문가 / 💎 프리미엄’처럼 만들어줘.

전문가, 일반회원, 기업, 프리미엄, 무료, 남자, 여자, 활성, 대기, 결제완료, 미납처럼 반복되는 단어는 텍스트를 계속 쓰지 말고 이모지나 배지로 줄여줘. 단, 너무 어수선하지 않게 핵심 반복 정보에만 적용해줘.

회원 선택 전에는 테이블이 화면 전체를 꽉 채우고, 회원 행이나 셀 어느 곳을 클릭해도 테이블이 줄어들면서 오른쪽에 상세 패널이 열리게 해줘. 오른쪽 상세 패널에는 기본정보, 상담/매칭, 결제/수수료, 활동/이력, 메모 탭과 2단/4단 정보 그리드를 넣어줘.

회원가입, 과정 신청서, 문의/상담 신청서는 한 줄에 많은 항목을 무리하게 넣지 말고 같은 성격의 항목끼리 묶어서 보기 편하게 만들어줘. 신청서는 신청분류, 신청자 정보, 과정 요약, 학습 목표, 전체동의 순서로 정리해줘. 전체동의는 기본 체크 상태이고, 접힌 상태에서는 제목만 보이게 하고, 펼치기 했을 때만 약관 본문이 보이게 해줘.

버튼은 필요 없는 것만 제외하고 추가, 수정, 삭제, 저장, 닫기, 더보기, 상태변경, 등급변경, 상담변경, SMS, 이메일, 내보내기 같은 관리 버튼을 기본 노출해줘.
```
